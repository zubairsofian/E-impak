# Security Specification - E-Impak PdP

## Data Invariants
- An impact record must have a valid subject, class, and date.
- The `formal_diary` is generated by AI and should be a non-empty string.
- `students_involved` must be an array of strings.
- All documents must have a `timestamp`.

## The "Dirty Dozen" Payloads (Rejected)
1. Missing `date`
2. `subject` is not a string
3. `formal_diary` exceeds 5000 characters
4. `students_involved` is not an array
5. `students_involved` array size exceeds 100
6. Missing `timestamp`
7. `timestamp` is not `request.time`
8. `raw_note` is too long ( > 2000 chars)
9. Spoofer attempting to write without auth
10. Spoofer attempting to delete a record they didn't create (future: need ownerId)
11. State injection: Adding a `verified: true` field not in schema
12. Huge document ID poisoning

## Security Rules Plan
- Default deny
- isSignedIn() helper
- isValidImpact() validation helper
- allow list/get: if isSignedIn() (In a real app, I'd filter by teacher/admin)
- allow create: if isSignedIn() && isValidImpact()
- allow delete: if isAdmin() (simplified)
